01 — The Problem

Why does Spring exist at all?

Before Spring, every Java object created its own dependencies using new. This seems harmless at first — until your classes become a tangled web of creation logic that's impossible to test or maintain.

OrderService.java ❌ TIGHT COUPLING 
public class OrderService {
  private PaymentService ps;
  private InventoryService inv;

  public OrderService() {
    // ❌ Creating its own deps
    ps  = new PaymentService();
    inv = new InventoryService();
  }
}
OrderService.java ✅ SPRING WAY 
public class OrderService {
  private final PaymentService ps;
  private final InventoryService inv;

  // ✅ Spring passes them in
  public OrderService(
      PaymentService ps,
      InventoryService inv) {
    this.ps  = ps;
    this.inv = inv;
  }
}
🔒
Hard to test
Can't swap a real PaymentService for a mock in tests
🕸
Dependency hell
One class change ripples through everything that creates it
♻️
No reuse
Every class creates its own copy — no shared singleton instances
02 — The Principle

Inversion of Control

The core idea of Spring is one sentence:
"Don't call us — we'll call you."

Instead of your objects building their dependencies, you hand that control over to Spring. Spring becomes the boss of object creation and wiring.

❌ Traditional
You're in control

Your class decides when and how to create PaymentService. It knows everything about its construction.

// Caller creates the callee
new PaymentService()
INVERT
IoC
✅ Spring IoC
Spring is in control

Spring decides when and how to create every object. Your class just declares what it needs and Spring delivers it.

// Spring creates & injects
@Autowired PaymentService ps;
DI — The Mechanism

IoC is the principle.
Dependency Injection is how it's done.

DI means Spring "injects" (passes in) the objects a class needs — either through the constructor, a setter method, or directly into a field. You declare what you need; Spring finds it and delivers it.

Constructor Injection ⭐ preferred
Setter Injection

03 — Beans

What is a Bean?

A Bean is simply any Java object whose lifecycle is managed by Spring. Nothing magical. Not a special class. Just a regular object that you've handed ownership of to Spring.

🏭

Spring Creates It

You never call new yourself. Spring reads your config and instantiates the object for you.

instantiation
📦

Spring Stores It

By default, Spring creates exactly one instance (singleton) and keeps it in the container, ready to inject.

singleton scope
🔌

Spring Wires It

Spring injects the bean wherever it's needed — no manual lookup, no new, no service locator pattern.

dependency injection
⏰

Spring Manages Lifecycle

From creation, through @PostConstruct init callbacks, to @PreDestroy cleanup on shutdown.

lifecycle
📖

Has a Name

Every bean has an ID (usually the class name in camelCase). Spring uses it to look up and inject the right instance.

bean id
🧬

Has a Scope

Singleton (default), Prototype (new instance each time), Request, Session — you control how many instances exist.

scopes

3 Ways to Define Beans

Choose your preferred style — Spring understands all three. 

@Configuration  // "This class defines beans"
public class AppConfig {

    @Bean  // "This method's return value is a bean"
    public PaymentService paymentService() {
        return new PaymentService();
    }

    @Bean
    public OrderService orderService() {
        // Spring intercepts this call — returns the SAME instance
        return new OrderService(paymentService());
    }
}
@Component  // "Spring, manage me as a bean"
public class PaymentService { ... }

@Component
public class OrderService {

    private final PaymentService paymentService;

    @Autowired  // "Inject the matching bean here"
    public OrderService(PaymentService ps) {
        this.paymentService = ps;
    }
}

@Configuration
@ComponentScan("com.myapp")  // Scan this package
public class AppConfig { }
<!-- applicationContext.xml (old-school way) -->
<beans>

    <bean id="paymentService"
          class="com.app.PaymentService"/>

    <bean id="orderService"
          class="com.app.OrderService">
        <constructor-arg ref="paymentService"/>
    </bean>

</beans>

// You'll see this in old codebases.
// Annotations / Java Config are modern standard.

05 — The Full Picture

How it all flows together

From your configuration file to a fully assembled, running application — here's the exact sequence Spring follows every time you start your app.

1

You write your Config

Whether it's a @Configuration class, component-scanned packages, or XML — you declare your beans.

@Configuration
public class AppConfig { @Bean ... }
2

ApplicationContext reads Config

You bootstrap the container by passing your config. Spring reads it and builds a map of "what beans do I need to create?"

ApplicationContext ctx =
  new AnnotationConfigApplicationContext(AppConfig.class);
3

Beans are instantiated

Spring creates instances of all beans — in the right order, resolving dependencies first before creating dependents.

// Spring creates:
PaymentService ← first (no deps)
OrderService ← second (needs PaymentService)
4

Dependencies are injected

Spring wires everything together — constructor args, setter calls, or field injection. Each bean gets exactly what it declared.

// Spring calls:
new OrderService(paymentService);
// orderService is now fully wired ✅
5

Lifecycle callbacks run

Any methods marked @PostConstruct are called — great for initialization logic after injection is complete.

@PostConstruct
public void init() {
  // runs after all deps injected
}
6

Your app runs

You call ctx.getBean() (or in Spring Boot, it's all automatic). The full object graph is assembled and ready.

OrderService os = ctx.getBean(OrderService.class);
os.placeOrder(99.99); // 🚀
7

Shutdown & cleanup

When the context closes, Spring calls @PreDestroy on beans — useful for closing connections, flushing buffers, etc.

@PreDestroy
public void cleanup() {
  // runs before bean is destroyed
}
06 — Mental Models

The analogies that make it click

Sometimes the best way to deeply understand a concept is through the right analogy. Carry these with you.

📞
Beans
Entries in a phonebook

Spring has a phonebook of every object it manages. When something needs a PaymentService, Spring looks it up and hands back the one it has on file.

👔
IoC
Spring is the boss

You're a worker. You don't find your own tools — the boss assigns them to you before your shift starts. Your job is just to do the work, not manage the supply room.

🖥️
Dependency Injection
Tools on your desk

Before you start working, someone puts everything you need right on your desk. You didn't go find them — they were delivered. Constructor injection = delivered before you sit down.

🧠
ApplicationContext
The phonebook + factory + manager

It knows what objects exist (phonebook), can create new ones on demand (factory), and tracks the whole lifecycle from birth to shutdown (manager). It's Spring's entire brain.

Now you understand Spring

Every single thing in Spring — AOP, Transactions, Security, MVC, and all of Spring Boot — is just more beans being added to this same container. Spring Boot auto-configures them for you.
