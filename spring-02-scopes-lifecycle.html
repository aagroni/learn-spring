01 — Scope

How many instances does Spring create?

When Spring creates a bean, a critical question arises: how many instances of this class should exist? One shared instance for the whole app? A fresh one every time someone asks? One per web request? This is what scope controls.

@Scope("singleton") ⭐

Singleton

One instance per ApplicationContext. Every time someone asks Spring for this bean, they get the exact same object back. This is the default — you don't even need to write the annotation.

02 — Lifecycle

The complete life of a Bean

From the moment Spring reads your config to the moment the application shuts down, a bean passes through a precise sequence of stages. Understanding this lets you hook in at exactly the right moment — for initialization, validation, connection setup, and cleanup.

1
Phase 1 — Instantiation

Spring calls the constructor

Spring uses reflection to call your bean's constructor (or factory method). At this point, the object exists but has no properties set — it's a blank slate.

// Spring does this internally:
OrderService bean = new OrderService();
2
Phase 2 — Dependency Injection

Properties & dependencies are injected

Spring populates all @Autowired fields, setters, and constructor args. After this step your bean has all its collaborators wired in.

// Spring wires collaborators:
bean.paymentService = ctx.getBean(PaymentService.class);
3
Phase 3 — Aware Interfaces

Spring injects infrastructure references

If your bean implements special *Aware interfaces, Spring calls their setter methods now — giving the bean a reference to the container itself, its own name, the classpath, etc.

// Example — bean wants its own name:
implements BeanNameAware {
  setBeanName("orderService"); // called by Spring
}
4
Phase 4 — BeanPostProcessor (Before)

Post-processors run before init

Spring calls all registered BeanPostProcessor.postProcessBeforeInitialization() methods. This is how Spring's own features like AOP proxies and @Transactional get woven in.

// Spring's AOP, @Transactional, etc. hook here
processor.postProcessBeforeInitialization(bean, beanName);
5
Phase 5 — Initialization ⭐ Your code hooks in here

Your initialization code runs

This is the step you'll use most often. Spring calls your init logic in this order: @PostConstruct method → InitializingBean.afterPropertiesSet() → custom initMethod.

@PostConstruct // ← use this one
public void init() {
  // All deps injected. Safe to use them here.
  cache.warmUp();
  connection.verify();
}
6
Phase 6 — BeanPostProcessor (After)

Post-processors run after init

Spring calls postProcessAfterInitialization(). This is where AOP creates the final proxy wrapper around your bean — the object placed in the container may actually be a proxy at this point.

// The bean in the container might now be a proxy:
// OrderService$$SpringProxy@4a2f ← wraps your real bean
7
Phase 7 — Ready & In Use

Your bean lives its life

The bean is now stored in the ApplicationContext and injected wherever needed. It serves requests, processes data, runs business logic — this is the longest phase, lasting the entire application lifetime (for singletons).

// Your app does its thing...
orderService.placeOrder(order);
orderService.cancelOrder(id);
8
Phase 8 — Destruction ⭐ Your cleanup code runs here

Cleanup before death

When the ApplicationContext closes (app shutdown, test teardown), Spring calls your cleanup logic: @PreDestroy method → DisposableBean.destroy() → custom destroyMethod.

@PreDestroy // ← use this one
public void cleanup() {
  // Called before Spring destroys this bean
  connection.close();
  scheduler.shutdown();
}
03 — Aware Interfaces

When beans need to talk back to Spring

Normally, your beans don't know they're inside Spring — and that's good design. But occasionally a bean legitimately needs infrastructure info: its own name, a reference to the context, the classloader. Spring provides Aware interfaces for exactly this.

BeanNameAware

Spring tells the bean its own name (the id it was registered with in the container).

setBeanName(String name)

ApplicationContextAware

Spring passes a reference to the ApplicationContext itself. Rarely needed — prefer injection instead.

setApplicationContext(ctx)

BeanFactoryAware

Gives the bean access to the underlying BeanFactory — useful for programmatic bean lookup.

setBeanFactory(factory)

EnvironmentAware

Provides access to the Environment object — properties, profiles, and system variables.

setEnvironment(env)
AwareExample.java 
@Component
public class AuditService implements BeanNameAware, EnvironmentAware {

    private String beanName;
    private Environment env;

    @Override
    public void setBeanName(String name) {
        this.beanName = name; // "auditService"
    }

    @Override
    public void setEnvironment(Environment env) {
        this.env = env;
    }

    @PostConstruct
    public void init() {
        // Both beanName and env are populated at this point
        String profile = env.getActiveProfiles()[0];
        log.info("{} starting in profile: {}", beanName, profile);
    }
}
04 — Your Toolkit

Lifecycle annotations you'll use daily

Most of the time, you only need two annotations. Forget the old interfaces — these are the clean, modern way to hook into a bean's lifecycle.

@PostConstruct

Runs after full injection

Called once, right after all dependencies are injected. Perfect for validation, cache warming, opening connections, loading initial data.

@PreDestroy

Runs before bean is destroyed

Called when the ApplicationContext closes. Perfect for releasing resources, closing connections, flushing queues, stopping scheduled tasks.

@Bean(initMethod=...)

External init method

When you don't own the class (3rd-party library), specify the init method name in the @Bean annotation instead.

@Bean(destroyMethod=...)

External destroy method

Similarly, specify the cleanup method name for 3rd-party classes. Spring calls it on shutdown — great for DataSource.close() etc.

DatabaseService.java — complete lifecycle example 
@Component
public class DatabaseService {

    private final DataSource dataSource;  // injected by Spring
    private Connection connection;

    // ① Constructor — called first, dataSource injected here
    public DatabaseService(DataSource dataSource) {
        this.dataSource = dataSource;
        // ⚠ DO NOT use dataSource here for complex init —
        // ⚠ other beans may not be ready yet
    }

    // ② @PostConstruct — called after full injection
    @PostConstruct
    public void init() {
        // ✅ Safe to use dataSource — everything is wired
        connection = dataSource.getConnection();
        connection.testQuery("SELECT 1");
        log.info("DB connection verified ✅");
    }

    // ③ Normal usage — serves the app
    public List<User> findAll() {
        return connection.query("SELECT * FROM users");
    }

    // ④ @PreDestroy — called on context close
    @PreDestroy
    public void cleanup() {
        if (connection != null) {
            connection.close();
            log.info("DB connection closed cleanly ✅");
        }
    }
}
05 — Mental Models

Analogies that make it stick

🏪
Singleton scope
The store's one cash register

Every customer uses the same register. There's exactly one. You don't build a new register for each customer — that would be wasteful and silly.

🎟
Prototype scope
Event tickets off the printer

Every time you ask for a ticket, a fresh one prints. Each one is unique, has its own state. Once printed, the printer doesn't track what happens to it.

👶
@PostConstruct
First day of school orientation

Before a student can attend classes (serve requests), they go through orientation — getting their schedule, ID, locker. All setup before the real work starts.

🧹
@PreDestroy
Clocking out at end of shift

Before leaving, you close open files, put tools back, and log your hours. You don't just walk out — you tidy up. @PreDestroy is your "end of shift" routine.

🏭
BeanPostProcessor
Quality control on the assembly line

Before a product ships, QC inspectors check it, add labels, apply coatings. BeanPostProcessors are Spring's QC line — wrapping beans in proxies before they're "shipped" to the container.

🪪
Aware interfaces
Employee badge with building access

Normally employees don't need to know which floor they're on. But security personnel get a special badge that grants them extra info about the building they're working in.

You now own the full lifecycle

Every Spring feature you'll use — transactions, security, caching, AOP — is implemented using BeanPostProcessors and this lifecycle. Once you see that, the magic disappears and real understanding takes its place.
