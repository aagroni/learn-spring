02 — REST Fundamentals

What REST actually means

REST (Representational State Transfer) is an architectural style for APIs over HTTP. It has one core rule: everything is a resource with a URL, and you interact with it using standard HTTP verbs. No SOAP envelopes, no RPC calls — just URLs, verbs, and JSON.

GET

Read a resource

Fetch one or a list. Never modifies data. Safe & idempotent — call it 100 times, same result.

safe + idempotent
POST

Create a resource

Creates a new resource. Calling it twice creates two resources. Returns 201 Created + Location header.

neither
PUT

Replace a resource

Full replacement — send the entire object. Idempotent: calling it twice produces the same result.

idempotent
PATCH

Partially update

Send only the fields you want to change. More efficient than PUT for large objects.

not guaranteed
DELETE

Remove a resource

Deletes the resource. Returns 204 No Content. Idempotent: deleting an already-deleted item is still OK.

idempotent
REST URL design — nouns not verbs, resources not actions 
// ❌ RPC-style (wrong)
GET  /getUsers
POST /createUser
POST /deleteUser?id=1

// ✅ REST-style (correct) — resources + HTTP verbs do the talking
GET    /users          // get all users
GET    /users/{id}     // get one user
POST   /users          // create a user (body has the data)
PUT    /users/{id}     // replace user completely
PATCH  /users/{id}     // update user partially
DELETE /users/{id}     // delete a user

// Nested resources
GET    /users/{id}/orders       // all orders of a user
GET    /users/{id}/orders/{oid} // specific order of a user
03 — Building Controllers

Your first REST Controller

A @RestController is just a Spring bean (Lecture 01) with two things: it handles HTTP requests mapped to its methods, and it automatically serializes return values to JSON. Everything else — validation, transactions, security — comes from the layers you already know.

UserController.java — a complete REST controller 
@RestController              // = @Controller + @ResponseBody
@RequestMapping("/users")     // base URL for all methods below
public class UserController {

    private final UserService userService;  // ← injected (Lecture 04)

    public UserController(UserService userService) {
        this.userService = userService;
    }

    // ── GET all ──
    @GetMapping
    public List<UserDto> getAll() {
        return userService.findAll();  // Jackson serializes to JSON array
    }

    // ── GET one ──
    @GetMapping("/{id}")
    public ResponseEntity<UserDto> getById(@PathVariable long id) {
        return userService.findById(id)
            .map(ResponseEntity::ok)           // 200 OK
            .orElse(ResponseEntity.notFound().build());  // 404
    }

    // ── POST create ──
    @PostMapping
    public ResponseEntity<UserDto> create(
            @RequestBody @Valid CreateUserRequest req) {
        UserDto created = userService.create(req);
        URI location = URI.create("/users/" + created.getId());
        return ResponseEntity.created(location).body(created);  // 201
    }

    // ── PUT full update ──
    @PutMapping("/{id}")
    public ResponseEntity<UserDto> update(
            @PathVariable long id,
            @RequestBody @Valid UpdateUserRequest req) {
        return ResponseEntity.ok(userService.update(id, req));
    }

    // ── DELETE ──
    @DeleteMapping("/{id}")
    public ResponseEntity<Void> delete(@PathVariable long id) {
        userService.delete(id);
        return ResponseEntity.noContent().build();  // 204
    }
}

04 — Request Journey

What happens when a request arrives

Every HTTP request travels through a precise pipeline before your controller method is called. Understanding this is essential for debugging and for adding cross-cutting concerns like auth, logging, or CORS.

1

HTTP Request hits Embedded Tomcat

Spring Boot starts an embedded Tomcat (or Netty/Jetty). It listens on port 8080 and receives the raw HTTP request from the client.

Tomcat / Servlet Container
2

Servlet Filters run (Security, CORS, Logging)

Before Spring even sees the request, Servlet Filters intercept it. Spring Security lives here — authentication happens before any controller code runs.

Filter Chain (OncePerRequestFilter)
3

DispatcherServlet receives the request

Spring's front controller — a single servlet that routes every request. Auto-configured by Spring Boot. Acts as the entry point into the Spring MVC layer.

DispatcherServlet (Spring's Front Controller)
4

HandlerMapping finds the right controller method

Spring scans all @RequestMapping annotations and finds the method whose URL and HTTP method match. GET /users/42 → getById(42).

RequestMappingHandlerMapping
5

HandlerInterceptors run (pre-handle)

Spring's equivalent of AOP for the web layer. Custom interceptors (auth checks, request logging, rate limiting) run here before the controller method is called.

HandlerInterceptor.preHandle()
6

Argument resolvers deserialize the request

Spring resolves @PathVariable, @RequestParam, and calls Jackson to deserialize the JSON body into your @RequestBody object.

HandlerMethodArgumentResolver + Jackson
7

@Valid triggers Bean Validation

If @Valid is present on a parameter, Hibernate Validator runs all constraint annotations (@NotNull, @Email, etc.). On failure: 400 Bad Request automatically.

Hibernate Validator / JSR-380
8

Your controller method executes

Finally, your actual code runs. The AOP proxy (Lecture 03) wraps it — so @Transactional, @PreAuthorize, and any custom aspects fire around your method.

Your @RestController method
9

Jackson serializes the return value to JSON

@RestController includes @ResponseBody — your returned object is automatically serialized to JSON by Jackson and written into the HTTP response body.

HttpMessageConverter + Jackson
10

Response sent back to client

HTTP status code, headers, and JSON body travel back through the filter chain and Tomcat to the client. Round trip complete.

HTTP Response → Client
05 — ResponseEntity & Status Codes

ResponseEntity — full control of the response

Returning a plain object from a controller always gives a 200 OK. ResponseEntity<T> gives you full control: choose the status code, set headers, and optionally include a body. Use it whenever the status code carries semantic meaning — which is almost always.

ResponseEntity — patterns you'll use every day 
// 200 OK with body
return ResponseEntity.ok(user);

// 201 Created with Location header (best practice for POST)
URI location = MvcUriComponentsBuilder.fromMethodName(
    UserController.class, "getById", created.getId()).build().toUri();
return ResponseEntity.created(location).body(created);

// 204 No Content (successful DELETE or update with no body)
return ResponseEntity.noContent().build();

// 404 Not Found
return ResponseEntity.notFound().build();

// 400 Bad Request with error body
return ResponseEntity.badRequest().body(new ErrorResponse("Invalid data"));

// Custom status + custom header
return ResponseEntity
    .status(HttpStatus.ACCEPTED)        // 202
    .header("X-Job-Id", jobId)
    .body(result);

HTTP Status Codes you must know

2xx
Success

The request was received and processed.

200 OK 201 Created 204 No Content 202 Accepted
3xx
Redirect

The resource has moved. Client should follow the Location header.

301 Moved Permanently 302 Found 304 Not Modified
4xx
Client Error

The client sent something wrong. Fix it on the client side.

400 Bad Request 401 Unauthorized 403 Forbidden 404 Not Found 409 Conflict 422 Unprocessable
5xx
Server Error

The server failed. The client's request was fine — something broke on your end.

500 Internal Server Error 502 Bad Gateway 503 Service Unavailable
06 — Validation & Error Handling

Never trust incoming data

Always validate what comes in. Spring integrates with Bean Validation (JSR-380) through Hibernate Validator. Add @Valid to your @RequestBody and annotate your DTO fields — Spring handles the rest. For global error shaping, use @ControllerAdvice.

CreateUserRequest.java — validated DTO 
public class CreateUserRequest {

    @NotBlank(message = "Name required")
    @Size(min=2, max=100)
    private String name;

    @Email(message = "Invalid email")
    @NotBlank
    private String email;

    @Min(18)
    @Max(120)
    private int age;

    @NotNull
    @Future
    private LocalDate startDate;

    // getters / setters
}
GlobalExceptionHandler.java — @ControllerAdvice 
@RestControllerAdvice  // catches exceptions globally
public class GlobalExceptionHandler {

    // Validation failure → 400
    @ExceptionHandler(MethodArgumentNotValidException.class)
    public ResponseEntity<ErrorResponse> handleValidation(
            MethodArgumentNotValidException ex) {
        List<String> errors = ex.getBindingResult()
            .getFieldErrors()
            .stream()
            .map(FieldError::getDefaultMessage)
            .toList();
        return ResponseEntity.badRequest()
            .body(new ErrorResponse("Validation failed", errors));
    }

    // Entity not found → 404
    @ExceptionHandler(EntityNotFoundException.class)
    public ResponseEntity<ErrorResponse> handleNotFound(
            EntityNotFoundException ex) {
        return ResponseEntity.notFound().build();
    }

    // Anything else → 500
    @ExceptionHandler(Exception.class)
    public ResponseEntity<ErrorResponse> handleGeneric(
            Exception ex) {
        return ResponseEntity
            .internalServerError()
            .body(new ErrorResponse("Something went wrong"));
    }
}
07 — Best Practices

REST API design rules that matter

These aren't opinions — they're the patterns that make APIs predictable, maintainable, and easy to consume. Junior devs who apply these stand out immediately.

Rule 01

Use DTOs, never expose entities

Never return your JPA @Entity directly from a controller. Use a DTO. Entities have lazy collections, bidirectional relationships, and hibernate proxies that cause infinite recursion or expose internal schema.

Rule 02

Version your API from day one

Start with /v1/users. When you need breaking changes, release /v2/users without killing existing clients. Impossible to add later without breaking things.

Rule 03

Always return consistent error shapes

Every error response should have the same structure: timestamp, status, message, errors[]. Clients need to be able to parse errors without guessing the schema.

Rule 04

Separate controller from service logic

Controllers only handle HTTP: parse request, call service, shape response. Business logic lives in @Service. This makes services testable without HTTP.

Rule 05

Use @Transactional in the service, not controller

You learned this in Lecture 05. Service layer owns transactions. Controllers own HTTP. Never blur this line — @Transactional on a controller method is a design smell.

Rule 06

Paginate list endpoints from the start

Never return unlimited lists. Use Spring Data's Pageable from day one. Adding pagination to an existing GET /users is a breaking change if clients aren't expecting it.

08 — Quick Reference

Annotations cheat sheet

@SpringBootApplication
Bootstrap the app

@Configuration + @ComponentScan + @EnableAutoConfiguration in one.

@RestController
JSON REST controller

@Controller + @ResponseBody. All methods serialize return values to JSON.

@RequestMapping
Base URL prefix

Set on the class to define the URL prefix shared by all methods inside.

@GetMapping / @PostMapping
Method-level routing

Shortcut for @RequestMapping(method=GET/POST). One annotation per verb.

@PathVariable
URL segment → param

Binds a {placeholder} from the URL path to a method parameter.

@RequestParam
Query string → param

Binds ?key=value from the query string. Supports defaultValue.

@RequestBody
JSON body → object

Jackson deserializes the HTTP request body into the annotated parameter.

@Valid
Trigger Bean Validation

Apply next to @RequestBody to run all JSR-380 constraint annotations.

@RestControllerAdvice
Global exception handler

Catches exceptions across all controllers. Shape your error responses here.

@ExceptionHandler
Handle specific exception

Inside @ControllerAdvice — maps a specific exception type to a response.

@CrossOrigin
Allow CORS

Permits cross-origin requests. Set origins and methods. Configure globally in WebMvcConfigurer.

@Value
Inject config property

Reads a value from application.properties into a field. Supports defaults.

09 — Mental Models

Analogies that make it stick

🤖
Spring Boot
A pre-configured laptop out of the box

Plain Spring is like buying all the parts and assembling the PC yourself. Spring Boot is like buying a MacBook — everything configured, drivers installed, ready to use. You can still open it up and customize.

📬
REST API
A post office with standard forms

Each URL is a PO box (resource). Each HTTP method is a standard form (GET=enquiry, POST=submit, DELETE=withdrawal request). Everyone uses the same forms — no custom formats.

🚦
DispatcherServlet
Airport traffic control

Every flight (request) goes through the same control tower first. It doesn't handle the flight itself — it looks at the destination and routes it to the right runway (controller).

🧱
@ControllerAdvice
The building's emergency response team

Every floor (controller) can have a fire, but there's one dedicated team that handles all emergencies the same way. No need for every floor to set up their own fire protocol.
