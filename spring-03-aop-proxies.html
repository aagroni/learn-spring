AOP & Proxies

01 — The Problem

The cross-cutting concern problem

Imagine you need to add logging, transaction handling, and security checks to 20 service methods. You could copy-paste those 3 concerns into every method — but now your business logic is buried under infrastructure noise, and changing "how we log" means touching 20 files. This is the cross-cutting concern problem.

❌ WITHOUT AOP — business logic polluted
OrderService.java 
public Order placeOrder(OrderDto dto) {
  // infrastructure noise ↓
  log.info("Entering placeOrder...");
  if (!security.isAuthorized()) throw...;
  transaction.begin();

  // actual business logic ↓ (3 lines!)
  inventory.reserve(dto);
  Order o = orderRepo.save(dto);

  // infrastructure noise ↓
  transaction.commit();
  log.info("Exiting placeOrder...");
  return o;
}
✅ WITH AOP — pure business logic
OrderService.java 
@Transactional      // ← AOP handles this
@PreAuthorize("...")  // ← AOP handles this
@LogExecutionTime   // ← your custom AOP
public Order placeOrder(OrderDto dto) {
  // pure business logic — nothing else
  inventory.reserve(dto);
  return orderRepo.save(dto);
}
02 — The Mechanism

Spring AOP works through Proxies

Spring AOP is not a compiler trick or bytecode manipulation. It is beautifully simple: when you put an AOP annotation on a bean, Spring doesn't give you your real object back from the container. It gives you a proxy — a generated subclass that wraps your object and intercepts every method call. This is why you saw in Lecture 02 that a bean might end up as OrderService$$SpringCGLIB.

The proxy intercepts the call → runs all matching advice (in order) → delegates to your real method → runs post-advice → returns result.
Your real object never knows any of this is happening.
JDK Dynamic Proxy

Used when your bean implements an interface. The proxy implements the same interface. Fast, built into Java, no extra library needed. 

// Spring creates a proxy that
// implements OrderService interface
interface OrderService { ... }
class OrderServiceImpl implements OrderService
CGLIB Proxy (default in Spring Boot)

Used when your bean has no interface. CGLIB generates a subclass at runtime. Spring Boot defaults to CGLIB even for interface-based beans. 

// Spring generates a subclass:
class OrderService$$CGLIB
  extends OrderService {
  // overrides every method
}
03 — The Vocabulary

5 words you must know

AOP has its own vocabulary. These 5 terms map to real things in your code — once they click, reading AOP configuration is effortless.

Aspect

The class that holds your cross-cutting logic

Your LoggingAspect, SecurityAspect — a regular class annotated with @Aspect that contains all the advice for one concern.

Pointcut

The "where" — which methods to intercept

An expression that matches method signatures. "All methods in the service package", "any method annotated with @Transactional". This is how Spring knows which calls to intercept.

Advice

The "what" — code that runs at the pointcut

The actual logic that executes. Could be before the method, after it, around it, or only on exception. Think of it as the interceptor code itself.

JoinPoint

The "when" — a specific method invocation

A single, concrete moment during execution — like "this specific call to placeOrder() right now". It's the runtime object your advice receives, containing method args, target object, etc.

Weaving

The process of applying aspects

The act of connecting aspects to application objects. Spring does runtime weaving via proxies (at context startup). Other AOP frameworks can weave at compile time or load time.

04 — Advice Types

When does your advice run?

There are 5 advice types — each controlling exactly when your interceptor code executes relative to the target method. @Around is the most powerful and used most in practice.

@Around — full control

Wraps the entire method call. You decide whether to even call the real method, what arguments to pass, and what return value to give back. Most powerful — used by @Transactional, @Cacheable.

USE WHEN: timing, caching, retry logic, transaction management, modifying args or return value 
@Aspect @Component
public class LoggingAspect {

  @Around("execution(* com.app.service.*.*(..))")
  public Object logTime(ProceedingJoinPoint pjp)
      throws Throwable {

    long start = System.currentTimeMillis();

    Object result = pjp.proceed(); // ← call real method

    long elapsed = System.currentTimeMillis() - start;
    log.info("{} took {}ms",
      pjp.getSignature().getName(), elapsed);

    return result;
  }
}

@Before — runs before the method

Executes before the target method is called. Cannot prevent the method from executing (use @Around for that). Cannot modify the return value.

USE WHEN: input validation, logging entry, security checks (Spring Security uses @Around though) 
@Aspect @Component
public class AuditAspect {

  @Before("execution(* com.app.service.*.*(..))")
  public void logEntry(JoinPoint jp) {
    String method = jp.getSignature().getName();
    Object[] args = jp.getArgs();
    log.info("→ Entering {} with args: {}",
      method, Arrays.toString(args));
  }
}

@After — runs after, always

Executes after the method finishes — whether it succeeded or threw an exception. Like a finally block. Cannot access the return value.

USE WHEN: cleanup, releasing resources, logging exit regardless of outcome 
@Aspect @Component
public class CleanupAspect {

  @After("execution(* com.app.service.*.*(..))")
  public void cleanup(JoinPoint jp) {
    // Runs regardless of success or exception
    // like a finally block
    log.debug("← Exiting {}",
      jp.getSignature().getName());
    ThreadLocalcontext.clear();
  }
}

@AfterReturning — success only

Runs only when the method completes normally (no exception). You get access to the actual return value through the returning attribute.

USE WHEN: logging successful results, post-processing return values, sending events on success 
@Aspect @Component
public class EventAspect {

  @AfterReturning(
    pointcut = "execution(* com.app.service.OrderService.placeOrder(..))",
    returning = "order"
  )
  public void onOrderPlaced(Order order) {
    // Only runs on success, gets the return value
    eventBus.publish(new OrderPlacedEvent(order));
  }
}

@AfterThrowing — exceptions only

Runs only when the method throws an exception. You get the exception object. Useful for centralized error handling and monitoring — does NOT suppress the exception.

USE WHEN: error logging, metrics on failures, alerting on specific exceptions 
@Aspect @Component
public class ErrorTrackingAspect {

  @AfterThrowing(
    pointcut = "execution(* com.app.service.*.*(..))",
    throwing = "ex"
  )
  public void trackError(JoinPoint jp, Exception ex) {
    // Only on exception — doesn't suppress it
    metrics.increment("errors." + jp.getSignature().getName());
    alerting.notify(ex);
  }
}
Pointcut Expression Syntax — the patterns you'll use daily 
// ① All methods in a package
"execution(* com.app.service.*.*(..))"
//  ↑return  ↑package    ↑class ↑method ↑any args

// ② Any method annotated with @Transactional
"@annotation(org.springframework.transaction.annotation.Transactional)"

// ③ Methods in a specific class
"execution(* com.app.service.OrderService.*(..))"

// ④ Combine with && || !
"execution(* com.app.service.*.*(..)) && !execution(* com.app.service.*.find*(..))"

// ⑤ Reusable named pointcut
@Pointcut("execution(* com.app.service.*.*(..))")
public void serviceLayer() {} // reference as serviceLayer() in advice
05 — Execution Order

What happens in which order

// orderService.placeOrder(dto) — full execution trace with @Around + @Before + @AfterReturning
1
proxy intercepts

Controller calls proxy, not your real object

Spring injected a proxy. The controller has no idea. It calls placeOrder() on what looks like OrderService but is actually the CGLIB wrapper.

2
@Before advice runs

Pre-method interceptors fire

Any @Before advice matching this pointcut runs. Logs entry, validates input, etc. The real method hasn't been called yet.

3
@Around advice enters

Around advice code runs up to pjp.proceed()

The @Around advice runs all code before the pjp.proceed() call — starting a timer, opening a transaction, checking cache.

4
your real method

pjp.proceed() — your actual code runs

Finally, your real placeOrder() business logic executes. This is the only part you wrote — everything else was woven around it.

5
@Around returns

Around advice code after pjp.proceed()

Control returns to the @Around advice. It records elapsed time, commits the transaction, stores result in cache, then returns the result.

6
@AfterReturning / @After

Post-method advice fires

@AfterReturning (on success) or @After (always) runs last — publishing events, final logging, cleanup.

06 — The Gotchas

Where AOP silently breaks

These are the bugs that confuse even experienced Spring developers. Know them now and save yourself hours of debugging.

Self-invocation doesn't work

If method A inside a bean calls method B in the same bean with this.methodB(), AOP won't intercept methodB — because the call never goes through the proxy.

Inject the bean into itself using @Autowired and call through it, or use ApplicationContext to get the proxy.

Private methods are never intercepted

Proxies can only override public and (CGLIB) protected methods. Private methods are invisible to the proxy — @Transactional on a private method silently does nothing.

Always put AOP annotations on public methods. Spring even warns you in logs.

final classes/methods can't be proxied

CGLIB generates a subclass. If your class or method is final, it can't be subclassed — Spring will throw an error or silently skip AOP.

Don't mark Spring beans as final. Kotlin classes are final by default — use open or the Kotlin Spring plugin.

@Transactional on new Thread loses context

Transactions are bound to the current thread. If you spawn a new thread inside a @Transactional method, the new thread has no transaction — any DB calls it makes are outside the transaction.

Use Spring's @Async with proper propagation, or pass data across thread boundaries explicitly.
07 — Quick Reference

AOP annotations cheat sheet

@Aspect
Marks an Aspect class

Tells Spring this class contains advice. Also needs @Component.

@Pointcut
Reusable pointcut

Names an expression so multiple advice methods can reference it without duplication.

@Around
Full method control

Must call pjp.proceed() to invoke the real method. Most flexible.

@Before
Pre-method

Runs before. Can't stop the method or see the return value. Simple and fast.

@After
Always after

Runs after, success or exception. Like a finally block.

@AfterReturning
Success only

Access the return value via the returning param.

@AfterThrowing
Exception only

Access thrown exception via the throwing param.

@EnableAspectJAutoProxy
Enable AOP

Put on a @Configuration class. Spring Boot does this automatically.

Analogies that make it real

🕵️
Proxy
A body double that runs errands first

Your real celebrity is in the building, but everyone talks to the body double. The double does the security check, signs NDAs, then passes you through to the real person.

City-wide traffic cameras

You define: "record every car that enters downtown." Every car (method) matching that description gets recorded — the cars themselves don't know the camera exists.

🥪
@Around
The sandwich around your filling

Your real method is the filling. @Around advice is the bread on both sides — you control what's on top, what's on the bottom, and whether the sandwich even gets made.

🏛️
AOP in Spring
Laws that apply to every business

Tax law, fire safety, labor law — they apply to every business without each business writing their own compliance code. @Transactional, @PreAuthorize work the same way.

The magic is just a proxy

Now when @Transactional silently rolls back a transaction, or @PreAuthorize blocks an unauthorized call, you know exactly what's happening: a CGLIB subclass intercepted that method call and ran advice around it. No magic. Just objects.
