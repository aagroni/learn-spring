01 - The Problem

JPA is the bridge between objects and tables

Your application thinks in objects: User, Order, Product. Your database stores rows in tables. JPA is the mapping layer that keeps those two worlds aligned. Spring Data JPA then adds a cleaner repository API so most CRUD code disappears.

JPA

The specification

A Java standard that defines how entities, relationships, persistence context, and queries should work.

Hibernate

The implementation

The most common JPA provider. It does the real work: dirty checking, SQL generation, caching, lazy loading.

Spring Data JPA

The productivity layer

Repository interfaces, derived queries, pagination, sorting, and integration with Spring transactions.

Database

The source of truth

PostgreSQL, MySQL, H2, MariaDB, Oracle, SQL Server. JPA talks SQL through JDBC under the hood.

02 - Entity Mapping

An entity is a Java class mapped to a table

The entity is not just a DTO with annotations. It represents persistent state. JPA tracks changes to managed entities inside the persistence context, then synchronizes those changes with the database during flush or commit.

Product.java ENTITY 
@Entity
@Table(name = "products")
public class Product {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(nullable = false, length = 120)
    private String name;

    @Column(nullable = false, precision = 10, scale = 2)
    private BigDecimal price;

    @Enumerated(EnumType.STRING)
    private ProductStatus status;

    protected Product() {
        // JPA needs a no-arg constructor
    }

    public Product(String name, BigDecimal price) {
        this.name = name;
        this.price = price;
        this.status = ProductStatus.ACTIVE;
    }
}

Entity rules that save pain

  • Use a nullable wrapper type like Long for generated IDs.
  • Prefer BigDecimal for money, never floating point.
  • Use EnumType.STRING, because ordinal values break when enums change.
  • Keep entities focused on persistence and domain behavior, not HTTP request/response shape.
  • Do not expose entities directly from REST controllers. Map them to DTOs.
03 - Repositories

A repository is a typed gateway to persistence

Extend JpaRepository<Entity, IdType> and Spring generates the implementation at runtime. You get CRUD, batch operations, sorting, pagination, and query method parsing without writing boilerplate SQL.

ProductRepository.java SPRING DATA 
public interface ProductRepository
        extends JpaRepository<Product, Long> {

    Optional<Product> findByNameIgnoreCase(String name);

    Page<Product> findByStatusOrderByNameAsc(
        ProductStatus status,
        Pageable pageable
    );

    Page<Product> findByPriceBetween(
        BigDecimal min,
        BigDecimal max,
        Pageable pageable
    );

    boolean existsByNameIgnoreCase(String name);
}
save

Insert or update

save(entity) persists new entities and merges detached ones. Use it intentionally.

findById

Returns Optional

Missing rows are normal. Model that with Optional instead of returning null.

delete

Removes rows

Deletion can trigger cascades. Know your relationships before deleting aggregate roots.

Pageable

Limit your lists

Return pages for collection endpoints. Unlimited findAll() is a future incident.

04 - Query Methods

Spring can read a method name like a query sentence

Derived queries are great for simple filters. When the method name starts becoming a paragraph, switch to @Query, specifications, Querydsl, or a custom repository method.

findBy

Start the query. Other prefixes: existsBy, countBy, deleteBy.

Status

Match the entity field name exactly. Spring parses Java properties, not database column names.

AndPriceLessThan

Add conditions with operators like Between, Containing, In.

OrderByNameAsc

Static sorting can be encoded in the method. Dynamic sorting should use Sort.

List<Product> findByStatusAndPriceLessThanOrderByNameAsc(ProductStatus status, BigDecimal price);
JPQL query 
@Query("""
    select p
    from Product p
    where p.status = :status
      and lower(p.name) like lower(concat('%', :term, '%'))
    order by p.name asc
""")
List<Product> searchActiveProducts(
    @Param("status") ProductStatus status,
    @Param("term") String term
);
Native SQL when needed 
@Query(
    value = """
        select *
        from products p
        where p.created_at > now() - interval '30 days'
        order by p.created_at desc
    """,
    nativeQuery = true
)
List<Product> findRecentlyCreated();
05 - Relationships

Relationships describe foreign keys, not JSON nesting

Entity relationships model how tables connect. They are powerful, but they can also create slow queries, accidental deletes, and infinite JSON recursion. Keep ownership clear and map responses with DTOs.

@ManyToOne
orders id PK customer_id FK
customers id PK

Many orders belong to one customer. This is usually the owning side with the foreign key.

@OneToMany
customer orders collection
order customer FK

Often the inverse side. Be careful with lazy loading and JSON serialization.

@ManyToMany
students_courses student_id FK course_id FK

Uses a join table. In real apps, often replace it with an explicit entity like Enrollment.

Order.java - relationship mapping 
@Entity
@Table(name = "orders")
public class Order {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @ManyToOne(fetch = FetchType.LAZY, optional = false)
    @JoinColumn(name = "customer_id", nullable = false)
    private Customer customer;

    @OneToMany(
        mappedBy = "order",
        cascade = CascadeType.ALL,
        orphanRemoval = true
    )
    private List<OrderLine> lines = new ArrayList<>();
}
06 - Service Layer

Transactions belong in the service layer

A transaction is a unit of work. Put @Transactional around service methods that change data or need consistent reads. The repository talks to the database; the service decides what the business operation means.

ProductService.java TRANSACTION 
@Service
public class ProductService {

    private final ProductRepository productRepository;

    public ProductService(ProductRepository productRepository) {
        this.productRepository = productRepository;
    }

    @Transactional
    public ProductResponse create(CreateProductRequest request) {
        if (productRepository.existsByNameIgnoreCase(request.name())) {
            throw new DuplicateProductException(request.name());
        }

        Product product = new Product(
            request.name(),
            request.price()
        );

        Product saved = productRepository.save(product);
        return ProductResponse.from(saved);
    }

    @Transactional(readOnly = true)
    public Page<ProductResponse> findActive(Pageable pageable) {
        return productRepository
            .findByStatusOrderByNameAsc(ProductStatus.ACTIVE, pageable)
            .map(ProductResponse::from);
    }
}
Managed

Inside persistence context

JPA tracks changes. If you change a field in a transaction, Hibernate can update it automatically.

Detached

Outside persistence context

The object still has data, but JPA is no longer tracking it. Changes are not flushed automatically.

Flush

SQL synchronization

Hibernate sends pending SQL before commit or before queries that require database consistency.

Commit

Database confirmation

The transaction succeeds and changes become durable. On rollback, the database is restored.

07 - Pitfalls

The bugs are usually in the invisible SQL

Spring Data JPA feels magical when it works. Senior developers keep the generated SQL in mind. Turn on SQL logging while learning, inspect slow endpoints, and treat relationships with respect.

N+1 queries

You load 1 list, then each row triggers another query for its related entity. Looks harmless in Java, slow in production.

Fix: fetch join, @EntityGraph, DTO projection

LazyInitializationException

A lazy relationship is accessed after the transaction closes. Hibernate cannot fetch it anymore.

Fix: fetch what you need in the service

Returning entities from controllers

Entities can expose internal fields, trigger lazy loading, or serialize recursive relationships forever.

Fix: response DTOs

Accidental cascade deletes

Cascade settings can delete more data than expected if ownership is modeled casually.

Fix: cascade only inside aggregates

Using findAll for APIs

It works with 20 rows. It hurts with 2 million rows. Collection endpoints should be paginated.

Fix: Pageable + indexes

Schema drift

Letting Hibernate auto-update production schema can create surprising migrations.

Fix: Flyway or Liquibase migrations
08 - Quick Reference

Annotations cheat sheet

@Entity
Persistent class

Marks a class as a JPA entity that maps to a database table.

@Table
Table metadata

Customizes table name, indexes, unique constraints, and schema.

@Id
Primary key

Marks the field that uniquely identifies each entity row.

@GeneratedValue
ID generation

Chooses how IDs are generated: identity, sequence, table, or auto.

@Column
Column rules

Controls name, nullability, length, precision, uniqueness, and more.

@Enumerated
Enum storage

Use STRING to store enum names instead of fragile numeric positions.

@ManyToOne
Many rows to one

Common owning side for a foreign key relationship.

@OneToMany
One row to many

Often inverse side. Combine with mappedBy to avoid extra join tables.

@JoinColumn
Foreign key column

Defines the column used to join one entity to another.

@Transactional
Unit of work

Defines the transactional boundary around a service operation.

@Query
Custom query

Write JPQL or native SQL when derived method names are not enough.

@EntityGraph
Fetch plan

Load selected relationships eagerly for a specific repository method.

Persistence is where Spring becomes real

REST gives your app a public surface. Spring Data JPA gives it memory. Once you understand entities, repositories, transactions, and generated SQL, you can build APIs that survive real data, real users, and real scale.
