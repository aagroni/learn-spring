01 — The Basics

What @Autowired actually does

When Spring creates a bean that has an @Autowired dependency, it searches the ApplicationContext for a matching bean and injects it. This search has a precise order — and understanding that order is what prevents nasty runtime surprises.

needs a bean
OrderService
Has a field / constructor param of type PaymentGateway
Spring resolves
injects this bean
StripeGateway
Found in context, implements PaymentGateway, marked @Primary
1

Match by Type

Spring first searches for all beans assignable to the required type. If there's exactly one match, it's injected. Done. No further steps needed.

2

Multiple matches → check for @Primary

If multiple beans match the type, Spring looks for one marked @Primary. If exactly one is primary, it wins. If multiple are primary — error.

3

Still ambiguous → check @Qualifier or name

Spring checks if a @Qualifier("name") is specified, or if the variable name matches a bean name. If it finds one match, inject it.

4

Still ambiguous → NoUniqueBeanDefinitionException 💥

Spring found multiple candidates and no way to disambiguate. App fails to start. You must use @Primary, @Qualifier, or a more specific type.

02 — Disambiguation Tools

@Primary vs @Qualifier

When you have multiple beans of the same type, you have two tools. @Primary sets a default. @Qualifier selects a specific one at the injection site. Both have their place.

@Primary — set the default
Two implementations, one primary 
// Two beans of same type
@Component
@Primary  // ← default for all injections
public class StripeGateway
    implements PaymentGateway { ... }

@Component
public class PayPalGateway
    implements PaymentGateway { ... }

// Injection — no @Qualifier needed
@Autowired
private PaymentGateway gateway;
// ↑ gets StripeGateway automatically
@Qualifier — pick a specific one
Select by name at injection site 
@Component("stripe")
public class StripeGateway
    implements PaymentGateway { ... }

@Component("paypal")
public class PayPalGateway
    implements PaymentGateway { ... }

// Inject a specific one
@Autowired
@Qualifier("paypal")
private PaymentGateway gateway;
// ↑ gets PayPalGateway specifically
Bonus — inject ALL beans of a type as a List 
// Spring injects every PaymentGateway bean into this list
@Autowired
private List<PaymentGateway> allGateways;
// allGateways = [stripeGateway, paypalGateway, ...]

// Useful for plugin patterns, chain of responsibility, etc.
03 — Injection Styles

3 ways to inject — one right choice

Spring supports three injection styles. They all work. But they're not equal. Constructor injection is the only style that produces provably correct, testable beans.

Constructor Injection ⭐ PREFERRED

Dependencies via constructor

All deps declared in the constructor. Bean is impossible to create without them. Immutable fields. Zero surprises. Testable without Spring.

✓ Always use this 
@Component
public class OrderService {
  private final PaymentService ps;

  // @Autowired optional if one constructor
  public OrderService(PaymentService ps) {
    this.ps = ps;
  }
}
Setter Injection — niche use

Dependencies via setters

Deps can be changed after construction. Useful for optional dependencies or when you need to break a circular dependency (but fix the design instead).

⚠ Only for optional deps 
@Component
public class ReportService {
  private Notifier notifier;

  @Autowired
  public void setNotifier(Notifier n) {
    this.notifier = n;
  }
}
Field Injection — avoid

Dependencies via @Autowired fields

Looks clean but hides dependencies. Impossible to test without Spring. Fields can't be final. Creates beans with invisible required state.

✗ Avoid in production code 
@Component
public class OrderService {
  @Autowired  // ← hidden dependency
  private PaymentService ps;

  // Can't be final
  // Hard to test without Spring
}
04 — The Circular Trap

Circular dependencies

A circular dependency happens when Bean A needs Bean B to be created, but Bean B needs Bean A to be created. Spring detects this and throws an error on startup — which is actually helpful because circular deps always signal a design flaw.

⚠ BeanCurrentlyInCreationException — circular dependency detected
ServiceA
↓ needs ↑ needs
ServiceB
Spring can't create A without B, can't create B without A → deadlock

How to fix it

The real fix is to rethink the design. Circular deps are a code smell — usually you need to extract a third class that both depend on. If you truly can't redesign:

Extract shared logic to a 3rd class ⭐ Use setter injection on one side @Lazy on one constructor param 
// Last resort: @Lazy breaks the cycle
public ServiceA(@Lazy ServiceB b) {
    this.b = b;
}
// Spring creates a proxy for ServiceB,
// resolves the real bean on first use
05 — Optional & Lazy

Handling optional and lazy dependencies

Optional deps, @Lazy, and @Value 
@Component
public class NotificationService {

    // ① Optional dep — won't fail if no SlackClient bean exists
    @Autowired(required = false)
    private SlackClient slackClient;

    // ② Java Optional — expressive, null-safe
    private final Optional<SlackClient> slack;
    public NotificationService(Optional<SlackClient> slack) {
        this.slack = slack;
    }

    // ③ @Lazy — bean not created until first use
    private final HeavyReportEngine engine;
    public NotificationService(@Lazy HeavyReportEngine e) {
        this.engine = e; // proxy until first method call
    }

    // ④ @Value — inject config properties
    @Value("${app.notifications.enabled:true}")
    private boolean enabled;
}
06 — Mental Models

Analogies that stick

🔍
Type matching
HR searching a job board by role

You need a "Backend Developer". HR searches all applicants by that role. One result → hired. Multiple → narrow by seniority (@Primary) or name (@Qualifier).

⭐
@Primary
The default supplier

Your company has 3 office supply vendors. One is marked "preferred." All orders go there by default, unless you specifically request a different vendor.

🏷
@Qualifier
Ordering from a specific branch

Same supplier, 3 branches. You put the branch name on your order form. Spring delivers from exactly that branch — no guessing.

🐍
Circular dependency
The snake eating its own tail

Two employees who can only start work after the other one finishes onboarding. Neither can start. It's a design problem, not a Spring problem.
