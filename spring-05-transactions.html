01 — The Concept

What a transaction actually is

A transaction groups multiple database operations into one atomic unit. Either all of them succeed and are committed to the database, or any failure causes all of them to roll back — leaving the database as if nothing happened. @Transactional tells Spring: "wrap this method in a transaction." That's it. The AOP proxy from Lecture 03 handles all the transaction mechanics around your method.

Transactions guarantee ACID properties

A
Atomic

All or nothing. Either every operation commits or none do. Half-done state is impossible.

C
Consistent

The database moves from one valid state to another. Constraints are never violated mid-transaction.

I
Isolated

Concurrent transactions don't see each other's in-progress changes (controlled by isolation level).

D
Durable

Once committed, the change survives crashes, power failures, system restarts.

What @Transactional actually does under the hood (AOP proxy) 
// What you write:
@Transactional
public void transferMoney(Account from, Account to, double amount) {
    accountRepo.debit(from, amount);
    accountRepo.credit(to, amount);
}

// What Spring's AOP proxy does around it:
public void transferMoney(...) {
    TransactionStatus tx = txManager.getTransaction(txDef); // begin
    try {
        // ↓ your actual code runs here
        accountRepo.debit(from, amount);
        accountRepo.credit(to, amount);

        txManager.commit(tx);       // all good → commit
    } catch (RuntimeException e) {
        txManager.rollback(tx);      // anything fails → rollback
        throw e;
    }
}
02 — Propagation

What happens when transactional methods call each other?

Propagation controls what Spring does with an existing transaction when a new @Transactional method is called. Should it join the outer transaction? Create a new one? Run without any? This is where most bugs hide.

Propagation Behavior Common use
REQUIRED DEFAULT Join existing transaction if one exists. Create new one if none. 99% of service methods. The default you want.
REQUIRES_NEW Always create a brand new transaction. Suspend any existing one. Audit logging — must commit even if outer tx rolls back.
SUPPORTS Run in transaction if one exists. Run without one if none. Read-only queries that are fine either way.
NOT_SUPPORTED Suspend any existing transaction. Always run non-transactionally. Operations that must not be in a transaction.
MANDATORY Join existing transaction. Throw exception if none exists. Methods that must only be called from within a transaction.
NEVER Run without transaction. Throw exception if one exists. Operations that must never run transactionally.
NESTED Create a nested transaction (savepoint). Roll back only nested part on failure. Partial rollback within a larger transaction.
REQUIRES_NEW in practice — audit log must always commit 
@Service
public class PaymentService {

    @Autowired private AuditService audit;

    @Transactional  // outer transaction
    public void processPayment(Payment p) {
        paymentRepo.save(p);

        audit.log(p);  // REQUIRES_NEW → own transaction
        // ↑ audit.log commits independently
        // ↑ even if processPayment rolls back, audit row is saved

        if (p.isInvalid()) throw new RuntimeException(); // rolls back payment
    }
}

@Service
public class AuditService {

    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void log(Payment p) {
        auditRepo.save(new AuditEntry(p));  // always committed
    }
}
03 — Isolation

Isolation — how much do concurrent transactions see?

When two transactions run simultaneously, isolation controls how much one can see of the other's in-progress changes. Higher isolation = more protection but lower performance (more locking).

READ_UNCOMMITTED — dangerous

Can read changes that haven't been committed yet ("dirty reads"). Fastest, but nearly never correct.

dirty reads non-repeatable reads phantom reads
READ_COMMITTED — common default

Only reads committed data. The default in most databases (PostgreSQL, Oracle). Prevents dirty reads.

✓ no dirty reads non-repeatable reads phantom reads
REPEATABLE_READ

Reading the same row twice in one transaction gives the same result. Prevents non-repeatable reads. Default in MySQL InnoDB.

✓ no dirty reads ✓ repeatable reads phantom reads
SERIALIZABLE — strongest

Transactions execute as if sequential. Full protection. Highest isolation but slowest — uses heavy locking.

✓ no dirty reads ✓ repeatable reads ✓ no phantom reads
Setting isolation level 
// Use database default (usually READ_COMMITTED)
@Transactional
public void normalOperation() { ... }

// Explicitly set isolation for critical financial ops
@Transactional(isolation = Isolation.SERIALIZABLE)
public void criticalBankTransfer() { ... }

// Read-only — Spring hint to database to optimize reads
@Transactional(readOnly = true)
public List<Order> findAllOrders() { ... }
04 — Rollback Rules

When does Spring rollback? This surprises everyone.

This is the most misunderstood rule in Spring transactions. By default, Spring only rolls back on RuntimeException (unchecked) and Error. Checked exceptions do NOT trigger rollback by default.

❌ Checked Exception — NO rollback by default

If your method throws IOException, SQLException, or any custom checked exception, Spring will commit the transaction unless you configure otherwise. This is a dangerous default that bites everyone. 

@Transactional
public void processFile() throws IOException {
    userRepo.save(user);
    throw new IOException("file missing");
    // ⚠ user IS COMMITTED despite the exception!
}

✅ Fix — declare rollback for checked exceptions

Use rollbackFor to explicitly add checked exceptions to the rollback rule. Many teams just use rollbackFor = Exception.class as a blanket rule. 

@Transactional(rollbackFor = Exception.class)
public void processFile() throws IOException {
    userRepo.save(user);
    throw new IOException("file missing");
    // ✅ user is rolled back now
}

// Or: rollback some, not others
@Transactional(
  rollbackFor = IOException.class,
  noRollbackFor = FileNotFoundException.class
)
05 — readOnly

readOnly = true — always use it on reads

For any method that only reads data, mark it @Transactional(readOnly = true). Spring passes this hint to the database driver and JPA, unlocking several optimizations.

Class-level default + method-level override pattern 
@Service
@Transactional(readOnly = true)  // ← default for all methods in this class
public class ProductService {

    public List<Product> findAll() { ... }         // readOnly=true ✅
    public Optional<Product> findById(long id) { ... } // readOnly=true ✅

    @Transactional  // ← override: write method, readOnly=false
    public Product createProduct(ProductDto dto) { ... }

    @Transactional
    public void updatePrice(long id, double price) { ... }
}
// readOnly=true benefits:
// ① Hibernate skips dirty-checking (no need to track changes)
// ② Some DBs route reads to read replicas
// ③ Explicit intent in code — documents "this never writes"
06 — The Gotchas

Silent bugs that destroy production

These are the exact mistakes that cause data corruption in production. Every Spring developer eventually hits these. Hit them here instead.

Self-invocation — @Transactional ignored

Calling a @Transactional method from within the same bean via this.method() bypasses the proxy. The transaction annotation is silently ignored.

Inject the bean into itself with @Autowired and call through the injected reference. Or refactor to a separate bean.

Private method — @Transactional ignored

The AOP proxy can't intercept private methods. Putting @Transactional on a private method does absolutely nothing. No error, no warning — just no transaction.

@Transactional must be on public methods. If the method should be private, wrap its call in a public transactional method.

Checked exception → no rollback

As covered in rollback rules: throwing a checked exception (like IOException) does not roll back the transaction by default. Data gets committed on exception.

Add rollbackFor = Exception.class or convert to a RuntimeException. Be explicit about your rollback rules.

Catching exceptions inside the method

If you catch and swallow a RuntimeException inside a @Transactional method, Spring never sees it — the transaction commits even though something went wrong.

Either let exceptions propagate, or if you must catch, call TransactionAspectSupport.currentTransactionStatus().setRollbackOnly() to force rollback.

@Transactional on @Controller

Technically works, but violates separation of concerns. Controllers should delegate to services. Transaction boundaries belong in the service layer, not the web layer.

Put @Transactional on your @Service classes. Controllers handle HTTP; services handle business logic and transactions.

Long transactions holding DB locks

A transaction that does heavy computation, calls external APIs, or sends emails while holding a DB connection is a performance disaster — it blocks other transactions from acquiring locks.

Keep transactions short. Call external services outside the transaction boundary. Do computation first, then open transaction for the DB writes.
07 — Full Flow

What Spring does on every @Transactional call

// Complete execution flow for a @Transactional(propagation=REQUIRED, rollbackFor=Exception.class) method
1
proxy intercepts

AOP proxy intercepts the method call

The TransactionInterceptor (a BeanPostProcessor from Lecture 02) is the advice running @Around your method. It handles everything below.

2
propagation check

Check for existing transaction

Spring checks if a transaction already exists on the current thread. Based on propagation setting, it either joins it, creates a new one, or suspends it.

3
begin transaction

Transaction opened, bound to thread

The transaction is started and bound to a ThreadLocal. This is how Spring ensures the same DB connection is used throughout — any JPA/JDBC code in this thread gets the same connection.

4
your code runs

pjp.proceed() — your method executes

Your actual business logic runs. Every save(), delete(), findById() uses the same transactional connection from the ThreadLocal.

5a
success path

No exception → commit

All DB operations are flushed to the database as a single atomic commit. Connection is released back to the pool.

5b
failure path

Exception matches rollback rules → rollback

All DB operations are undone. The database is restored to its state before step 3. Connection released. Exception re-thrown to caller.

08 — Quick Reference

@Transactional attribute cheat sheet

propagation
Transaction joining

How to behave with existing transactions. Default: REQUIRED.

isolation
Concurrency level

How much concurrent transactions see of each other. Default: database default.

readOnly
Read hint

Hints DB and Hibernate to optimize for reads. Use on all query methods.

rollbackFor
Rollback trigger

Which exception types trigger rollback. Default: RuntimeException + Error only.

noRollbackFor
Rollback exclusion

Exception types that should NOT trigger rollback despite matching rollbackFor.

timeout
Max duration

Seconds before transaction is forcibly rolled back. Guards against long locks.

09 — Mental Models

Analogies that make it permanent

✍️
Transaction
Signing a contract in pencil

You make all your changes in pencil. Commit = sign in ink permanently. Rollback = erase everything back to the original blank paper.

🏦
ACID / Atomicity
Bank transfer: debit and credit together

Money leaving one account and arriving in another must be one atomic action. Half-done (money leaves but doesn't arrive) is worse than doing nothing.

🧵
ThreadLocal binding
One cashier, one till

Your transaction is tied to your thread like a cashier tied to their till. Every operation on that thread uses the same till — no mixing with other cashiers' transactions.

🪆
Propagation.REQUIRED
Nested meetings with the same agenda

If you're called into a meeting already in progress, you join it — you don't start a separate meeting. All decisions are made together; one cancellation cancels all.
